## 1. 架构概览

### 1.1 架构风格
**单体应用 + 微服务治理组件**：单 JAR 部署，但集成 Nacos/Sentinel/Seata/RocketMQ 全套 Spring Cloud Alibaba 组件，预留微服务拆分能力。

## 3. 中间件依赖

| 中间件 | 端口 | Docker镜像 | 用途 |
|--------|:--:|------|------|
| MySQL 8.0 | 3306 | `mysql:8.0` | 主数据库, InnoDB引擎 |
| Redis Stack | 6380 | `redis/redis-stack:latest` | 向量存储 + 缓存 + 分布式锁 |
| RedisInsight | 8002 | (Redis Stack内置) | Redis可视化管理 |
| Elasticsearch | 9200 | `elasticsearch:8.11.0` | 全文检索, 中文分词 |
| Nacos | 8848 | `nacos/nacos-server:v2.2.3` | 服务注册与发现 |
| RocketMQ | 9876 | `apache/rocketmq:5.1.4` | NameServer, 异步消息 |
| Seata | 8091 | - | 分布式事务 (当前disabled) |

### 中间件启动顺序

```
1. MySQL  →  等待健康检查通过
2. Redis  →  等待 PING 响应
3. Nacos + RocketMQ + Elasticsearch  →  并行启动
4. 应用后端  →  最后启动
```

## 4. 部署架构


## 5. 关键技术决策

### 5.1 为何是单体而非微服务？

- 当前业务规模不需要微服务的复杂度
- 但引入 Nacos/Sentinel 预埋微服务治理能力
- 未来可按业务域拆分：用户服务/商品服务/订单服务/AI服务

### 5.2 为何 LangChain4j 而非 Spring AI？

- Spring AI 当时不够成熟，类缺失问题
- LangChain4j 对 OpenAI 兼容接口支持更好
- 提供 `ContentRetriever`、`AiServices` 等开箱即用的 RAG 组件

### 5.3 为何 Redis 做向量库？

- 复用现有 Redis 基础设施（向量存储 + 缓存 + 锁 三位一体）
- Redis Stack 原生支持向量索引 (FLAT + COSINE)
- 1536维 Embedding 在 Redis 中检索性能: ~5ms (1000条)

### 5.4 为何虚拟线程被禁用？

```java
System.setProperty("spring.threads.virtual.enabled", "false");
```
- JDK 21 虚拟线程与某些库(如 Netty)拧在一起时存在已知 Pin 问题
- 使用传统线程池 (core 50, max 200)，性能已满足当前需求

## 6. 安全架构

```
请求 → CorsFilter → AuthInterceptor(JWT) → Sentinel → Controller
         │              │                      │
         ▼              ▼                      ▼
     允许所有源     解析Bearer Token        流量控制
     (开发模式)    白名单路径跳过           熔断降级
                   7天过期
```

| 安全层 | 实现 |
|--------|------|
| 传输安全 | HTTPS (生产环境Nginx终结) |
| 认证 | JWT Bearer Token, Base64编码 |
| 鉴权 | AuthInterceptor 拦截器 |
| 密码存储 | BCrypt ($2b$10$...) |
| AI API Key | 环境变量, 不暴露前端 |
| SQL注入 | MyBatis-Plus 参数化查询 |
| 敏感词 | `sensitive-word` 库过滤AI输入 |
| 限流 | Redis滑动窗口 + Sentinel |
